lasif.project¶

Project management class.

It is important to not import necessary things at the method level to make importing this file as fast as possible. Otherwise using the command line interface feels sluggish and slow. Import things only the functions they are needed.

_get_coordinates_for_waveform_file(waveform_filename, waveform_type, network_code, station_code, event_name)[source]¶

Internal function used to grab station coordinates from the various sources.

Used to make sure the same functionality is used everywhere.

Parameters:
  • waveform_filename – The absolute filename of the waveform.
  • waveform_type – The type of waveform file. Could be deduced from the pathname but probably not necessary as the information is likely readily available in situations where this method is called. One of “raw”, “processed”, and “synthetic”
  • network_code – The network id of the file. Same reasoning as for the waveform_type.
  • station_code – The station id of the file. Same reasoning as for the waveform_type.
  • event_name – The name of the event. Same reasoning as for the waveform_type.

Returns a dictionary containing “latitude”, “longitude”, “elevation_in_m”, “local_depth_in_m”. Returns None if no coordinates could be found.

_validate_event_files(ok_string, fail_string, flush_point, add_report)[source]¶

Validates all event files in the currently active project.

The following tasks are performed:
  • Validate against QuakeML 1.2 scheme.
  • Check for duplicate ids amongst all QuakeML files.
  • Make sure they contain at least one origin, magnitude and focal mechanism object.
  • Some simply sanity checks so that the event depth is reasonable and the moment tensor values as well. This is rather fragile and mainly intended to detect values specified in wrong units.
  • Events that are too close in time. Events that are less then one hour apart can in general not be used for adjoint tomography. This will naturally also detect duplicate events.

Returns a dictionary containing a little bit of information about all stations available for a given event.

An available station is defined as a station with an existing station file and an existing waveform file.

Will return an empty dictionary if nothing is found.

Example return value:

{"BW.ROTZ": {"latitude": 10, "longitude": 11, "elevation_in_m": 12,
             "local_depth_in_m": 13},
 "BW.ROTZ2": ...,
 ...}
Parameters:

get_waveform_data(event_name, station_id, data_type, tag=None, iteration_name=None)[source]¶

Get an ObsPy Stream representing the requested waveform data.

This is the only interface that should ever be used to read waveform data. This assures a consistent handling of various issues.

Synthetic data will be properly rotated in case the project is defined for a rotated domain.

Parameters:
  • event_name (str) – The name of the event.
  • station_id (str) – The id of the station in question.
  • data_type (str) – The type of data to retrieve. Can be either raw, processed, or synthetic.
  • tag (str) – If requesting processed data, the processing tag must be given.
  • iteration_name (str) – If requesting synthetic data, the iteration name must be given.
Return type:

obspy.core.stream.Stream
Returns:

An up to three-component Stream object containing the requested data.

validate_data(station_file_availability=False, raypaths=False, waveforms=False)[source]¶

Validates all data of the current project.

This commands walks through all available data and checks it for validity. It furthermore does some sanity checks to detect common problems. These should be fixed.

Event files:
  • Validate against QuakeML 1.2 scheme.
  • Make sure they contain at least one origin, magnitude and focal mechanism object.
  • Check for duplicate ids amongst all QuakeML files.
  • Some simply sanity checks so that the event depth is reasonable and the moment tensor values as well. This is rather fragile and mainly intended to detect values specified in wrong units.
